# 코딩 스타일 (비표준 규칙만)

## 불변성 (필수)
- 항상 새 객체/배열 생성, 절대 뮤테이션 금지
- spread operator, map, filter 등 불변 패턴 사용

## 파일 구성
- 작은 파일 여러 개 > 큰 파일 소수
- 함수: 50줄 이내, 파일: 800줄 이내
- 기능/도메인별로 구성 (타입별 아님)
- `components/`, `hooks/`, `utils/` 같은 타입별 덤프 폴더를 기본 구조로 쓰지 않는다.
- 파일은 변경 이유가 같은 것끼리 묶는다. 함께 바뀌지 않는 코드는 같은 파일에 두지 않는다.
- 공통 모듈 승격은 3회 이상 반복되고, 이름 붙일 수 있는 도메인 개념일 때만 한다.

## 분리 구조
- 외부 입력/API DTO, DB schema, domain model, UI state를 같은 타입으로 공유하지 않는다.
- boundary에서 parse/validate한 뒤 내부 domain shape로 정규화한다.
- API response shape를 UI 전역에서 직접 전달하지 않는다. 필요한 view model로 변환한다.
- `utils`는 도메인 없는 순수 함수만 허용한다. 도메인 규칙은 해당 feature/domain 아래에 둔다.
- 비슷해 보여도 도메인이 다르면 합치지 않는다. 중복 제거보다 소유권 명확성을 우선한다.

## 단순화
- 한 번만 쓰이는 helper는 인라인한다.
- 2회 반복은 유지 가능하다. 3회 이상 반복부터 추출을 검토한다.
- 추상화는 호출부를 실제로 단순하게 만들 때만 둔다.
- 깊은 조건문은 guard clause, early return, 작은 pure function으로 낮춘다.
- 매직 값은 이름 있는 상수로 올리되, 한 파일에서만 의미가 있으면 파일 내부에 둔다.

## 정규화
- 같은 의미의 데이터를 여러 shape로 들고 다니지 않는다.
- nullable/optional 상태는 boundary에서 정리하고 내부에서는 가능한 한 명시적 상태로 표현한다.
- enum/string union/status 값은 한 곳에서 정의하고 변환 함수를 둔다.
- 날짜, 금액, ID, 권한 같은 값은 문자열 그대로 흘리지 말고 의미 있는 타입/validator를 둔다.

## 리팩토링
- 동작 변경과 구조 변경을 같은 커밋에 섞지 않는다.
- public API, props contract, DB schema, route contract 변경은 별도 변경으로 명시한다.
- 테스트 코드는 명시성이 우선이다. DRY를 이유로 의도를 숨기지 않는다.
- 리팩토링 후 타입체크, 테스트, 관련 lint를 다시 실행한다.
- HITL, 리뷰, PR 전에는 simplify gate를 확인하고 필요한 단순화/정규화를 끝낸다.
